# 数据核查流程（详细参考）

> 基于 2026-07-20/21 春季读书会资料包两轮实战提炼。
> 项目内 skill 方法，触发词："数据核查"。
> 定义见 `AGENTS.md`。

---

## Phase 1 — 只读扫描

**目标**：在不修改任何文件的前提下，全面了解数据现状。

### 步骤

1. 定位数据目录
   - 用户指定 → 使用用户说的路径
   - 没有指定 → 查找最近的分类子目录或包含 `.xlsx` 的目录

2. 读取可读文件
   - xlsx → openpyxl 读全部单元格（先看是否有公式，再读值）
   - txt / md / csv → 直接读
   - docx → python-docx 读取段落文本
   - png / jpg / pdf → 跳过（非文本可读文件）

3. 忽略的文件类型
   - `.git/`, `.agents/`, `.codex/` 等隐藏目录
   - `node_modules/`, `__pycache__/`, `.venv/` 等依赖目录
   - `*_修复版.*` 等已生成的产物文件
   - `chart_data.json`, `package-lock.json` 等非数据文件

4. 扫描时关注的 5 类问题
   - **字段不一致** — 同一字段在不同文件中命名、格式、取值不同
   - **金额单位不一致** — 数值列没有单位或单位不统一
   - **计算口径不明** — 数字的来源假设、人数基准、单价未经说明
   - **疑似超预算** — 状态"待定"的项目可能突破已知上限
   - **报告引用错误** — 复盘/结论与数据表无法交叉验证

5. 每个发现的输出格式
   ```
   **发现 N | 类别**
   - 文件：相对路径/文件名
   - 位置：具体位置
   - 原文：原文引用
   - 问题：一句话描述
   - 已处理：如有修复版则标注
   ```

## Phase 2 — 提问（每次 1 个）

**目标**：确认业务口径，而不是由 AI 猜测。

### 顺序策略

1. 第一个问题：最核心的数值单位或字段口径（例如"计划金额的单位是什么"）
2. 第二个问题：第二个关键字段口径（例如"物料数量的单位是什么"）
3. 第三个问题：版本/数据源问题（例如"哪个文件是正式版"）
4. 后续问题：根据之前的回答递进

### 每个问题的结构

```
**问题 N / 5**

我在 `文件路径` 的 `具体位置` 中看到：
[原文引用]

请问：[具体问题]

（附：如果字段有多种写法 → 问统一叫法
     如果数字不合理 → 问正常范围或例外条件
     如果缺失值不确定 → 问必填/可空/待补充）
```

### 未问到的发现

整理为"待确认清单"，在修复阶段列在 `检查日志.md` 的"待人工确认"部分。

### 不允许的行为

- 不自己判断口径
- 不一次问多个问题
- 不跳过提问直接修复

## Phase 3 — 修复

**目标**：生成可审查的修复版，让用户决定是否替换原文件。

### 修复规则

- 每个修复文件保存为 **同名 + `_修复版`** 后缀
- **不覆盖**任何原文件
- Excel 文件保留：公式、格式、所有 sheet、数据类型
- 文本文件（md/txt）：直接创建 `_修复版` 副本

### 输出文档（`outputs/`）

#### 检查日志.md

```
# 检查日志

## 一、已处理的修复项

| 项目 | 修复内容 |
|------|----------|
| ... | 具体改动 |

## 二、待人工确认的项

| 问题 | 需要确认的内容 |
|------|---------------|
| ... | ... |

## 三、关键指标速览

| 指标 | 当前值（修复版） | 上限 | 状态 |
|------|:--------------:|:----:|:----:|
| ... | ... | ... | ✅/👁 |
```

#### 对照清单.md

```
# 原文件与候选修复版对照清单

| 原文件 | 候选修复版 | 关键差异 | 需人工确认 |
|--------|-----------|----------|:----------:|
| ... | ... | ... | 是/否 |
```

#### 自检清单.md

```
# 数据自检清单

每项：
1. 检查内容
2. 合格标准
3. 严重程度（🔴 必须通过 / 🟡 建议通过 / ⚪ 可选）
4. 需要人工确认的例外情况
```

---

## Phase 4 — 归档与提交（扩展可选）

经两轮实战后可选扩展：

1. 将修复版和输出文档归入 `outputs/YYYY-MM-DD_作业整理归档/`
2. 按标准模板生成打卡提交记录
3. 脱敏后通过 GitHub PR 提交作业报告
