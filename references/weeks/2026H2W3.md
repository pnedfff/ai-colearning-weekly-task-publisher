# 2026H2W3｜数据处理：用 Codex 按你的标准检查、修复

## 群公告

本周任务：准备一份可安全练习的数据副本，让 Codex 先看懂数据、再向你确认业务口径，把你的判断标准整理成质检清单，最后按清单输出问题清单和候选修复版。

这周最重要的不是“让 AI 自动改表”，而是练会一套可控的数据核查流程：只读扫描、确认口径、形成标准、按标准检查、人工确认、另存修复版。

最后至少得到四样东西：`自检清单.md`、`问题清单.md`、候选修复版文件和 `检查日志.md`。原始文件必须保留不动。

没有合适数据的同学，可以使用课程测试数据：<https://drive.weixin.qq.com/s?k=ADQABgdrAAkkVXscQ2>。也可以使用自己制作的脱敏 CSV、Excel 或公开数据。

## 任务目标

把“帮我检查一下数据”变成一套按自己业务标准执行的核查流程。

本周练习三个核心能力：

- 让 Codex 先只读扫描，不直接修改数据。
- 由人确认字段、单位、正常范围、缺失值和例外条件等业务口径。
- 让 Codex 按确认后的清单检查并生成候选修复版，最终是否采用仍由人判断。

## 参与者流程

1. 准备安全的数据副本
   - 可以使用课程测试数据、公开数据或自己制作的脱敏数据。
   - 真实工作数据必须先复制一份测试副本，不能直接操作唯一原件。
   - 删除姓名、电话、邮箱、客户名称、账号、订单号等敏感信息。

2. 让 Codex 只读扫描
   - 先识别 CSV、Excel 和说明文档中的字段、单位、公式及跨文件引用。
   - 列出字段不一致、单位不一致、口径不明、疑似异常和引用错误。
   - 每个发现都要标明文件名、具体位置和原文。
   - 这一阶段不修复、不生成新文件、不覆盖源文件。

3. 回答 5 个关键口径问题
   - 让 Codex 只问最影响检查结果的 5 个问题。
   - 由你确认统一字段名、允许值、金额单位、正常范围、缺失值规则和业务例外。
   - 不确定的内容标记为“待确认”，不要让 Codex 猜。

4. 生成质检标准
   - 把确认过的口径写入 `自检清单.md`。
   - 每项包含检查内容、合格标准、严重程度和需要人工确认的例外。
   - 这一步只生成清单，仍不检查或修复原文件。

5. 按清单检查数据
   - 让 Codex 逐项检查并生成 `问题清单.md`。
   - 每个问题包含严重程度、文件、位置、原文、问题描述、建议修复方式和是否需人工确认。
   - 人工判断哪些是真问题、哪些是合理例外，并明确允许修复的项目。

6. 生成候选修复版
   - 只处理你已经确认的问题。
   - 每个结果另存为带 `_修复版` 后缀的新文件，绝不覆盖源文件。
   - Excel 要尽量保留公式、格式、工作表和数据类型。
   - 同时生成 `检查日志.md` 和原文件、修复版对照清单。

7. 人工抽查并打卡
   - 至少抽查 3 个修复点，核对原值、修复值、规则和业务含义。
   - 确认源文件仍在，候选修复版可追溯，日志与实际改动一致。
   - 只提交脱敏后的过程报告，不上传原始数据或候选修复数据。

## 可复制提示词

只读扫描：

```text
请先只读扫描当前项目中的数据文件。

要求：
1. 不修复、不生成新文件、不覆盖任何原文件。
2. 忽略隐藏目录、依赖文件和已生成产物。
3. 列出你发现的字段不一致、金额单位不一致、计算口径不明、疑似超预算、报告引用错误等问题。
4. 每个发现都要写清楚文件名、位置和原文。
5. 扫描完成后，不要直接下结论，先准备向我提问。
```

确认业务口径：

```text
请根据刚才的只读扫描结果，先问我最影响检查结果的 5 个口径问题。

提问要求：
1. 只问 5 个，别一次性全问。
2. 每个问题都要带上你看到的原文和具体位置。
3. 如果字段有多种写法，问我统一叫法。
4. 如果数字看起来不合理，问我正常范围或例外条件。
5. 如果缺失值不确定，问我是必填、可空，还是待补充。
6. 不要自己判断，不要自己修复。
```

生成自检清单：

```text
请把我们刚才确认的口径整理成“自检清单.md”。

每个检查项都要包括：
1. 检查内容。
2. 合格标准。
3. 严重程度：必须通过 / 建议通过 / 可选。
4. 需要人工确认的例外情况。

只生成清单，不检查、不修复文件。
```

生成问题清单：

```text
请按“自检清单.md”逐项检查所有数据文件，输出“问题清单.md”。

每个问题必须包含：
1. 严重程度。
2. 所在文件。
3. 具体位置。
4. 原文。
5. 问题描述。
6. 建议修复方式。
7. 是否需要人工确认。

仍然不要修改原文件。完成后等待我确认哪些问题允许修复。
```

生成候选修复版：

```text
请只根据我已经确认的问题和修复方案，生成候选修复版文件。

要求：
1. 每个修复文件保存为新文件，文件名加“_修复版”后缀。
2. 不覆盖任何原文件。
3. Excel 文件要尽量保留公式、格式、工作表和数据类型。
4. 关键指标、实际改动和业务例外写入“检查日志.md”，标注需要人工抽查的项目。
5. 修复完成后，输出原文件与候选修复版的对照清单。
```

## 完成标准

只要提交内容能证明下面这些点，就算完成：

- 使用测试副本、公开数据或课程测试数据，没有直接操作唯一原件。
- 第一轮是只读扫描，没有直接修改或覆盖源文件。
- 回答了最影响结果的 5 个业务口径问题。
- 生成了包含合格标准、严重程度和例外条件的 `自检清单.md`。
- 按清单生成了可追溯到文件、位置和原文的 `问题清单.md`。
- 人工确认问题后才生成候选修复版，所有修复结果另存新文件。
- 生成了 `检查日志.md` 和原文件、修复版对照清单。
- 至少人工抽查 3 个修复点，并记录一次人工判断或纠正。
- 作业报告已脱敏，只提交过程和经验，不上传真实业务数据。

不强制使用 Excel，也不强制处理真实工作数据。用一份小型 CSV 或课程测试数据跑通完整流程即可。

## 带班复盘

审核打卡时，重点看三件事：

- 好的提交：先确认标准再检查，问题可以溯源，源文件与候选修复版清楚分开。
- 需要改进：直接让 AI 改表、没有确认口径、把候选修复版当成最终数据、没有人工抽查。
- 最适合群里讨论的问题：哪一个异常看起来像错误，最后却因为业务口径被判定为合理例外？

周复盘可以统计：

1. 大家最常遇到的数据问题类型。
2. 最常需要人工确认的业务口径。
3. AI 最容易误判的异常和最有价值的检查规则。
